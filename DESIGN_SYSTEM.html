Design System - Tela Formulário Geral

Sistema documentado a partir da tela principal e da tela de cadastro/edição de passageiro. O objetivo é preservar a linguagem real do formulário: operacional, densa, clara e sem ilha visual.

Princípios

O sistema nasce do produto real, não de uma landing page.

Uso diario
1. Operacional primeiro

A interface existe para criar, revisar e salvar serviços com pouco atrito. Densidade e previsibilidade valem mais que decoração.

2. Passageiro pertence ao formulário

Seleção, cadastro, edição, foto e duplicidade usam a mesma casca visual: campos claros, borda sutil, foco azul e estados fortes.

3. Estado e comportamento são design

Vazio, incompleto, erro, foco, hover, bloqueado e salvo precisam ser documentados. Não são detalhe cosmético.

Fazer
  • Usar tokens reais de styles.css.
  • Manter campos compactos e alinhados.
  • Usar modal do sistema para decisão crítica.
  • Usar verde apenas para ação positiva.
Não fazer
  • Não criar componente premium isolado.
  • Não trocar o formulário por hero/landing.
  • Não usar confirm() do navegador.
  • Não destacar valor de select com peso/cor própria.

Feedback

Toasts e avisos precisam usar as classes reais do formulário. Não criar alerta paralelo.

Mensagens
Toast stack
Passageiro adicionado.
Habilite a edição antes de alterar.
Resolva os campos obrigatórios.
Contrato
  • Stack fixa: .toast-stack, canto inferior direito.
  • Mensagem: .toast-message, texto curto e acionável.
  • Estados: .success, .warning, .error.
  • Fechar: .toast-close com aria-label.
  • Toast flutuante usa .toast-floating só para contexto ancorado.

Padrões de fluxo

Use esses contratos antes de criar componente novo.

UX
Agendamento principal
  1. Preencher agenda, cliente, solicitante e rota.
  2. Selecionar passageiros pelo picker.
  3. Editar endereço por passageiro ou usar endereço único.
  4. Validar faturamento, retorno e repetição antes de salvar.
Cadastrar passageiro
  1. Nome, cliente, idioma e classificação são obrigatórios.
  2. Telefone precisa validar país + número.
  3. Foto é opcional, mas deve ter preview/limpar.
  4. Duplicidade exige revisão antes de criar novo.
Edição de passageiro
  1. Abrir pelo nome do passageiro.
  2. Começar em modo leitura.
  3. Ao habilitar, salvar campo a campo.
  4. Sinalizar saving, saved, error e empty.

Acessibilidade

O formulário precisa funcionar com teclado, leitura de tela e foco visível.

WCAG 2.2
Checklist
  • Todo campo obrigatório usa label visível e aria-required quando aplicável.
  • Foco precisa usar --shadow-focus.
  • Modal usa role="dialog" e aria-modal="true".
  • Icon button precisa de aria-label.
  • Erro precisa texto, não apenas cor.
Responsivo
  • Desktop: menu lateral fixo e conteúdo em painel.
  • Até 1240px: tabs horizontais.
  • Até 760px: topbar compacta, marca escondida, ações reduzidas.
  • Campos mantêm largura estável com min(), flex-basis e tokens.

Definition of done visual:
1. Usa tokens reais.
2. Não quebra DOM/IDs/data-* existentes.
3. Funciona em desktop, tablet e mobile.
4. Tem foco visível.
5. Estado vazio/erro/salvando documentado.
6. Se for fonte publicada: npm run wr.
